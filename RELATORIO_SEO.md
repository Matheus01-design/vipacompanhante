# Relatório de Correções SEO — VipAcompanhante

**Data:** 2026-04-22
**Repo:** Matheus01-design/vipacompanhante
**Stack:** Next.js 14 (App Router) + Supabase + MercadoPago
**Domínio:** https://www.vipacompanhante.com

---

## Resumo executivo

Após migração do WordPress para Next.js, milhares de páginas pararam de ser indexadas pelo Google. Auditoria identificou **5 problemas críticos** e várias melhorias secundárias.

A causa raiz **não era o sitemap nem o robots.txt** — era arquitetural:

1. Um `layout.tsx` duplicado renderizava `<html lang="en"><title>Next.js</title>` dentro de TODAS as páginas de perfil individual.
2. As páginas mais importantes do site (perfil, busca por característica, homepage) eram **Client Components** sem renderização no servidor — o bot do Google recebia HTML praticamente vazio.
3. Não havia `robots.txt` apontando para o sitemap.
4. Existia rota duplicada para o mesmo perfil (conteúdo duplicado).
5. Configuração de trailing slash gerava redirects 308 em todos os links internos.

Este relatório documenta cada correção aplicada.

---

## ✅ Ação 1 — Removido layout.tsx duplicado de /acompanhante

**Arquivo removido:** `src/app/acompanhante/layout.tsx`

**O que tinha de errado:**

```tsx
// ANTES — código removido
export const metadata = {
  title: 'Next.js',
  description: 'Generated by Next.js',
}

export default function RootLayout({ children }) {
  return (
    <html lang="en">
      <body>{children}</body>
    </html>
  )
}
```

Este arquivo era um boilerplate padrão do `create-next-app` que ficou esquecido. Como o Next.js compõe layouts em cascata, ele estava:

1. **Renderizando um segundo `<html>` dentro do RootLayout** → HTML inválido (browsers escondem, mas crawlers ficam confusos).
2. **Sobrescrevendo o `<title>` do RootLayout** com "Next.js" em todas as páginas de perfil individual (`/acompanhante/[slug]`).
3. **Mudando `lang="pt-BR"` do RootLayout para `lang="en"`** → Google interpretava as páginas como inglês, quebrando o ranking em buscas em português.

**Impacto:** Como o site pode ter milhares de perfis, **provavelmente todos eles estavam com title = "Next.js"** no HTML servido ao Google. O bot tipicamente descarta páginas assim como duplicadas de baixa qualidade.

**Por que só remover (não substituir):** O `RootLayout` em `src/app/layout.tsx` já fornece `<html lang="pt-BR">`, `<head>`, schema JSON-LD, GA e tudo mais. Não precisa de layout intermediário em `/acompanhante`.

**Resultado esperado:** A partir do próximo deploy + recrawl do Google, as páginas de perfil voltam a ter o title correto vindo do `generateMetadata` (que será adicionado na Ação 2).

## ✅ Ação 2 — Refatorada `/acompanhante/[slug]` para Server-Side Rendering

**Arquivos:**
- `src/app/acompanhante/[slug]/page.tsx` — reescrito como Server Component
- `src/app/acompanhante/[slug]/PerfilCliente.tsx` — novo Client Component (toda UI interativa)

**O que tinha de errado:**

A página de perfil era um Client Component (`'use client'`) que buscava os dados via Supabase **dentro de um `useEffect`**. Isso significa que:

1. O servidor renderizava só uma `<div>Carregando...</div>` no HTML inicial.
2. O Googlebot recebia esse HTML quase vazio e não enxergava nome, cidade, fotos, descrição.
3. Não havia `generateMetadata`, então o `<title>` herdava o do RootLayout ("VipAcompanhante") em todos os perfis.
4. Não havia schema markup do tipo `Person`, `BreadcrumbList` etc.
5. Sem `canonical`, `og:image` específico ou `og:title` por perfil — todo perfil compartilhava o mesmo OG.

**O que foi feito:**

### 2.1 — `page.tsx` virou Server Component

Agora ele:
- **Busca o perfil no servidor** via `getAcompanhantePorSlug()` (já existia em `lib/queries.ts`).
- **Implementa `generateMetadata`** que devolve `title`, `description`, `canonical`, `openGraph` (com a foto real do perfil como imagem) e `twitter` específicos para cada perfil. Usa `meta_title`/`meta_description` do banco se existirem; caso contrário, gera a partir de nome + idade + cidade + estado.
- **Renderiza dois blocos JSON-LD**:
  - `Person` com nome, descrição, imagem e endereço (bairro, cidade, estado).
  - `BreadcrumbList` com a hierarquia Início → Estado → Cidade → Nome.
- **Chama `notFound()`** se o slug não existir → Next.js devolve 404 real (importante: 404 ensina o Google a desindexar perfis removidos).
- **Embute `<PerfilCliente perfilInicial={perfil} />`** passando os dados como prop.

### 2.2 — Toda UI interativa migrada para `PerfilCliente.tsx`

O Client Component agora:
- **Recebe os dados do perfil já prontos via prop** (não faz mais o fetch principal). O HTML do servidor já é renderizado com nome, fotos, descrição, valores, etc. — exatamente o que o Google indexa.
- **Mantém toda a interatividade**: galeria com navegação, modal de fotos, favoritar, denúncia, compartilhar, registro de visualização.
- **Carrega dados secundários (avaliações, disponibilidade, favorito do usuário) em background** — isso não afeta SEO porque já é interação pós-load.
- **Removeu o `<Suspense>` wrapper** que escondia o conteúdo enquanto carregava — não é mais necessário porque os dados já chegam prontos.

**Por que essa abordagem (e não SSR puro):**

A página tem muita interatividade real (galeria com swipe, modais, favoritar, denúncia). Converter tudo para SSR puro forçaria reescrever a UX. A solução híbrida do Next.js — Server Component que renderiza HTML completo + Client Component que hidrata para interação — entrega o melhor dos dois mundos: o Google vê o conteúdo completo no HTML inicial, e o usuário tem a UX interativa.

**Resultado esperado:**
- Cada perfil agora tem `<title>`, `<meta description>`, `<link rel="canonical">`, `og:image` próprios.
- O HTML servido contém H1 com o nome, descrição, cidade — tudo indexável.
- Schema `Person` + `BreadcrumbList` aparecem em rich results do Google.
- Perfil removido devolve 404 real (não 200 com "carregando").

## ✅ Ação 3 — Criado `app/robots.ts`

**Arquivo criado:** `src/app/robots.ts`

**O que tinha de errado:**

Não existia nem `public/robots.txt` nem `src/app/robots.ts`. Sem `robots.txt`:
- O Google chega no site, lê o padrão "tudo permitido", mas **não sabe onde está o sitemap**.
- O bot precisa "descobrir" o sitemap por conta própria (em `/sitemap.xml`) — funciona mas é lento.
- Páginas privadas como `/minha-conta` e `/admin` ficavam expostas pro crawler, gastando crawl budget.

**O que foi feito:**

Criado `src/app/robots.ts` (convenção Next 14 — Next gera o `/robots.txt` automaticamente):

```ts
{
  rules: [{
    userAgent: '*',
    allow: '/',
    disallow: ['/admin', '/api', '/login', '/cadastro',
               '/recuperar-senha', '/minha-conta'],
  }],
  sitemap: 'https://www.vipacompanhante.com/sitemap.xml',
  host: 'https://www.vipacompanhante.com',
}
```

**Decisões:**

- **`allow: '/'`** — libera todo o resto, principalmente os perfis e páginas SEO.
- **`disallow`** das rotas de área logada e API — não tem nada útil pro bot indexar.
- **`sitemap`** explícito — o Google passa a descobrir o sitemap na primeira visita, em vez de adivinhar.
- **`host`** com `www.` — reforça o domínio canônico (já está consistente em todos os redirects e `BASE_URL`).

**Resultado esperado:** A próxima visita do Googlebot vê o `robots.txt` e descobre o sitemap imediatamente, acelerando o recrawl das páginas que ainda não foram indexadas.

## ✅ Ação 4 — Resolvida duplicação de URL para perfis

**Arquivos alterados:**
- `src/app/mulheres/[estado]/[cidade]/[bairro]/[slug]/page.tsx`
- `src/app/homens/[estado]/[cidade]/[bairro]/[slug]/page.tsx`
- `src/app/trans/[estado]/[cidade]/[bairro]/[slug]/page.tsx`

**O que tinha de errado:**

O mesmo perfil podia ser acessado por **duas URLs diferentes**:

1. `/acompanhante/maria-silva` (rota oficial, com toda a galeria/interatividade)
2. `/mulheres/sp/sao-paulo/centro/maria-silva` (rota legada, versão mais simples)

Ambas renderizavam HTML válido com status 200. Para o Google isso é **conteúdo duplicado clássico** — ele tem que escolher qual indexar, divide o sinal de ranking entre as duas, e frequentemente acaba não rankeando bem nenhuma.

A rota legada já tinha um `<link rel="canonical">` apontando para a oficial, mas canonical é só uma "sugestão" — o Google às vezes ignora se o conteúdo for diferente. O ideal é nem servir a página duplicada.

**O que foi feito:**

Substituído o conteúdo das três versões da rota legada (`mulheres`, `homens`, `trans`) por um redirect 301 (HTTP permanent) para a URL canônica:

```ts
import { permanentRedirect } from 'next/navigation'

export default function PerfilLegadoMulheres({ params }) {
  permanentRedirect(`/acompanhante/${params.slug}/`)
}
```

**Decisões:**

- **`permanentRedirect` (308/301)** em vez de `redirect` (307/302): comunica ao Google "essa URL mudou para sempre, transfira todo o sinal SEO".
- **Mantemos os arquivos** (não deletamos as rotas) porque o Google pode ainda ter URLs legadas no índice — sem eles, o crawler veria 404 em vez do redirect.
- **Trailing slash no destino** (`/acompanhante/${slug}/`) para casar com o `trailingSlash: true` do `next.config.js`, evitando um segundo redirect em cascata.

**Resultado esperado:** Em poucas semanas, o Google vai consolidar todo o sinal de ranking nas URLs `/acompanhante/[slug]/`, e remover do índice as URLs longas. Isso simplifica o sitemap e aumenta a autoridade das URLs canônicas.

## ✅ Ação 5 — Refatorada `/c/[slug]` (páginas SEO de característica) para SSR

**Arquivos:**
- `src/app/c/[slug]/page.tsx` — reescrito como Server Component
- `src/app/c/[slug]/CategoriaCliente.tsx` — novo, só para o "Ver mais"

**O que tinha de errado:**

As páginas `/c/loiras-sp`, `/c/com-local-rj`, `/c/verificadas-mg` etc. são as principais páginas de SEO long-tail do site (combinam característica + estado e podem chegar a centenas de URLs). Mas elas eram **Client Components com fetch via `useEffect`**:

- O Google chegava na página, lia HTML com "Carregando..." e nada de conteúdo.
- O título da página (`<h1>`) só aparecia depois do JS rodar.
- Não havia `generateMetadata` — todas elas herdavam o título genérico do RootLayout.
- Sem `canonical`, então `/c/loiras-sp` e `/c/loiras-SP` podiam ser tratadas como duplicadas.
- Sem schema `CollectionPage`/`BreadcrumbList`.

**O que foi feito:**

### 5.1 — `page.tsx` virou Server Component

- A função `parseSlug()` (que decodifica o slug em filtros) foi movida para o servidor.
- `generateMetadata()` agora gera `title`, `description`, `canonical` e OpenGraph específicos por slug — exemplo:
  - `/c/loiras-sp/` → `<title>Acompanhantes Loiras em São Paulo</title>`
  - `/c/com-local-rj/` → `<title>Acompanhantes com Local Próprio em Rio de Janeiro</title>`
- A primeira página (24 perfis) é **buscada no servidor** e renderizada já no HTML inicial. O Google vê os cards com nome, cidade, foto.
- Schema `CollectionPage` + `BreadcrumbList` em JSON-LD.
- **`robots.index = parsed.valido`**: se o slug não bate com nenhum filtro conhecido, devolve `noindex`. Antes, qualquer coisa em `/c/qualquercoisa` retornava 200 com lista vazia → o Google indexava milhares de URLs lixo. Agora elas voltam `noindex` (ou 404 quando o `parsed.valido` é falso, via `notFound()`).

### 5.2 — `CategoriaCliente.tsx` cuida só do "Ver mais"

- Recebe os 24 perfis iniciais como prop (vindos do servidor).
- Quando o usuário clica em "Ver mais", busca a próxima página direto no Supabase via client.
- **Trocou os `href="/acompanhante/${slug}"` por `href="/acompanhante/${slug}/"`** (com trailing slash) para casar com o `next.config.js` e evitar redirects 308 nos cliques.

**Resultado esperado:**
- Cada `/c/[slug]/` indexa com title/description próprios.
- Schema `CollectionPage` ativa rich snippets de listagem no Google.
- URLs de filtro inexistente (ex: `/c/banana-sp`) deixam de poluir o índice.

## ✅ Ação 6 — Refatorada homepage para Server-Side Rendering

**Arquivos:**
- `src/app/page.tsx` — reescrito como Server Component (wrapper SEO + fetch inicial)
- `src/app/HomeCliente.tsx` — novo, recebe os perfis prontos por prop

**O que tinha de errado:**

A homepage era um Client Component gigante que:
- Não tinha `generateMetadata` — usava só o título genérico do RootLayout (`"VipAcompanhante"`).
- Sem `canonical`, `og:image` ou `og:description` específicos.
- Buscava todos os perfis via `useEffect` — o HTML inicial servido ao Google não tinha nenhum card.
- O bot via "Carregando..." e nada de listagem.

**O que foi feito:**

### 6.1 — Novo `page.tsx` (Server Component)

- Define `metadata` específica:
  - **Title**: `"Acompanhantes no Brasil — Perfis Verificados em Todos os Estados"`
  - **Description**: foco em "maior plataforma" + cidades principais.
  - **Canonical**: `https://www.vipacompanhante.com/` (com www e barra final, casando com o `trailingSlash: true`).
  - **OpenGraph** completo.
- Faz **2 queries em paralelo no servidor**:
  - 24 perfis ordenados por plano + ID (a primeira página).
  - 6 perfis Super VIP destacados.
- Passa os dois conjuntos como prop pro `<HomeCliente />`.

### 6.2 — `HomeCliente.tsx` recebe perfis iniciais

- Renomeado de `HomePage` para `HomeCliente`.
- `useState(perfisIniciais)` em vez de `useState([])` — o estado inicial já é o conjunto vindo do servidor.
- `setCarregando(false)` por padrão — sem mais "Carregando..." no primeiro render.
- Toda a lógica interativa (filtros de cidade, geolocalização, stories, autocomplete, modais, "ver mais") foi mantida idêntica.

**Trade-off conhecido:**

Quando o cliente hidrata e o `useEffect` que reage a mudanças de filtros roda pela primeira vez, ele dispara uma busca duplicada (refaz a query inicial). É um custo pequeno de uma request extra ao Supabase no primeiro carregamento — aceito em troca da simplicidade da refatoração. Se quiser otimizar depois, basta adicionar um `useRef` flag pra pular o primeiro fetch.

**Resultado esperado:**
- Homepage tem title/description próprios e ranqueia melhor para "acompanhantes brasil", "acompanhantes" + nome de estado.
- Os 24 primeiros perfis aparecem no HTML inicial — Googlebot indexa nomes, cidades e fotos.
- Footer SEO com links para `/c/loiras-sp/`, `/c/morenas-rj/` etc. continua presente (descoberta de páginas long-tail).

## ✅ Ação 7 — Padronizado trailing slash em todos os links de perfil

**Arquivos alterados (19 ocorrências):**
- `src/app/HomeCliente.tsx` (3 — cards principais, super VIPs, link no story)
- `src/app/[sexo]/page.tsx`
- `src/app/mulheres/[estado]/page.tsx`, `.../[cidade]/page.tsx`, `.../[bairro]/page.tsx`
- `src/app/homens/[estado]/page.tsx`, `.../[cidade]/page.tsx`, `.../[bairro]/page.tsx`
- `src/app/trans/[estado]/page.tsx`, `.../[cidade]/page.tsx`, `.../[bairro]/page.tsx`
- `src/app/c/[slug]/CategoriaCliente.tsx`
- `src/components/StoriesCarrossel.tsx`
- `src/app/minha-conta/page.tsx`, `src/app/admin/page.tsx`
- `src/components/AdminAvaliacoes.tsx`, `src/components/AdminDenuncias.tsx`

**O que tinha de errado:**

O `next.config.js` define `trailingSlash: true` (com barra final em todas as URLs canônicas). Mas todos os `<Link>` do site apontavam para `/acompanhante/${slug}` (sem barra). O fluxo era:

1. Usuário/bot clica num card → vai para `/acompanhante/maria-silva`.
2. Next.js detecta a falta de barra → devolve `308 Permanent Redirect` para `/acompanhante/maria-silva/`.
3. Browser/bot segue o redirect.

Todo clique e todo crawl gastava 2 requests em vez de 1. Para o Googlebot, isso significa **metade do crawl budget jogado fora** — o que num site com milhares de perfis representa milhares de URLs que ele não chega a visitar por dia.

**O que foi feito:**

Substituído `href={\`/acompanhante/${X.slug}\`}` por `href={\`/acompanhante/${X.slug}/\`}` em **19 ocorrências** espalhadas por 17 arquivos. Verificação via grep confirma: zero links sem barra final restantes.

**Decisão:**

Mantida a configuração `trailingSlash: true` (não mudei o `next.config.js`). Motivo: o WordPress antigo usava trailing slash, então é provável que as URLs antigas indexadas no Google já tenham `/`. Mudar isso forçaria centenas de redirects extras. Manter a convenção e só corrigir os links é mais seguro.

**Resultado esperado:**
- Cada clique vira 1 request em vez de 2.
- Crawl budget do Google liberado para indexar mais perfis na mesma janela de tempo.
- Page speed melhora (Largest Contentful Paint na navegação interna).

## ✅ Ação 8 — Migrado sitemap para `app/sitemap.ts` e adicionados bairros

**Arquivos:**
- `src/app/sitemap.xml/route.ts` — REMOVIDO (estrutura antiga)
- `src/app/sitemap.ts` — NOVO (convenção Next 14)

**O que tinha de errado:**

O sitemap estava em `src/app/sitemap.xml/route.ts` (route handler manual). Combinado com `trailingSlash: true`, o Next.js servia em `/sitemap.xml/` (com barra final) — o Google espera `/sitemap.xml` sem barra. Isso causa um `308 Permanent Redirect` toda vez que o bot tenta acessar o sitemap.

Além disso, o sitemap antigo tinha problemas de cobertura:

- **URLs sem barra final** (ex: `/mulheres/sp` sem `/`) — também causavam 308 ao serem indexadas.
- **Não incluía URLs de bairro** (ex: `/mulheres/sp/sao-paulo/centro/`) — só estado e cidade.
- **Cidades hardcoded** — só estados estavam no array, e as cidades eram geradas só por padrões fixos. Cidades reais que existem no banco não entravam.
- **Não incluía `/planos/`**.

**O que foi feito:**

### 8.1 — Migração para a convenção do Next 14

Criado `src/app/sitemap.ts` que exporta `default async function sitemap(): Promise<MetadataRoute.Sitemap>`. O Next.js automaticamente:
- Gera o XML.
- Serve em `/sitemap.xml` (sem barra final, da forma que o Google espera).
- Adiciona `Content-Type: application/xml`.
- Permite tipagem completa (TypeScript valida estrutura, prioridades, frequências).

### 8.2 — URLs de cidade e bairro vindas do banco

Em vez de gerar combinações fixas, o sitemap agora:
- Busca todos os perfis ativos com `(estado, cidade, bairro, sexo)`.
- **Slugifica** cidade e bairro com a mesma lógica usada nas rotas (sem acento, lowercase, hífen).
- Gera **URLs reais** que o sistema sabe responder, evitando 404s no sitemap.

Exemplos de URLs novas que entraram:
- `/mulheres/sp/sao-paulo/jardins/`
- `/mulheres/rj/rio-de-janeiro/copacabana/`
- `/trans/sp/santos/centro/`

### 8.3 — Trailing slash em 100% das URLs

Toda URL listada termina com `/`, casando com o `trailingSlash: true` do `next.config.js`. Sem mais redirects 308 vindos do próprio sitemap.

### 8.4 — Estrutura preservada

Mantido tudo o que já funcionava:
- Páginas estáticas (home, cadastro, blog, luna, planos).
- Combinações característica + estado em `/c/...`.
- Perfis individuais com `lastmod` real do banco.
- Posts do blog.

**Resultado esperado:**
- Google descobre o sitemap mais rápido (via `robots.txt`) e processa sem redirects.
- Bairros agora aparecem como URLs candidatas a indexação — abre a oportunidade de rankear long-tail por bairro (tipo "acompanhantes em Copacabana").
- Sitemap reflete o estado real do banco em vez de combinações hardcoded.

## 🚀 O que fazer depois do deploy

1. **Build local primeiro**: `npm run build` pra garantir que nada quebrou.
2. **Deploy** (Vercel ou onde estiver hospedado).
3. **Google Search Console**:
   - Reenviar o sitemap em **Sitemaps** → adicionar `https://www.vipacompanhante.com/sitemap.xml`.
   - Em **Inspeção de URL**, testar 3-5 URLs de perfil (ex: `/acompanhante/maria-silva/`) → clicar em "Solicitar indexação".
   - Acompanhar o relatório de **Páginas indexadas** nas próximas 2-4 semanas — deve subir progressivamente.
4. **Monitorar**:
   - "Cobertura" no GSC deve mostrar redução de "Página alternativa com tag canônica adequada".
   - "Páginas com erro" deve cair (sem mais 308 em loop).
   - "Performance" → impressões e cliques devem crescer.

## 📝 Próximas otimizações (não feitas agora)

- **`generateStaticParams`** nas páginas mais visitadas (estados, cidades top) → ISR pra performance ainda melhor.
- **Trocar `<img>` por `<Image>` do Next** → Core Web Vitals (LCP, CLS) melhoram, ranking sobe.
- **Sitemap index** pra escalar acima de 50k URLs (limite por arquivo de sitemap).
- **Estrutura de redirects 301 das URLs WP antigas que ainda dão 404** (verificar via GSC quais URLs antigas estão recebendo tráfego e mapear).
- **Schema `Service` / `Offer`** nas páginas de perfil pra rich results de preço.
